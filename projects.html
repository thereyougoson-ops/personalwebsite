          01 · Creative web app · PWA gallery + Python media pipeline
          
Anatoly Sokoloff: A Visual Journey

          
A data-driven artist gallery with an AI cinemagraph pipeline.

          
how it stays reliable offline-capable PWA from static files · resumable pipeline generates each clip once, then replays it locally forever

        

        

          An installable PWA presenting a painter's 150+ work catalogue as an immersive, horizontally-scrolling
          timeline. A sibling Python pipeline turns selected paintings into looping AI cinemagraphs, so the gallery
          can play subtle motion with zero per-view inference cost.
        

        

          
The problem
A large body of work (150+ paintings across four series and decades) needed a narrative, installable experience rather than a flat grid, while staying trivial to host and free to animate.

          
The build
React SPA + PWA manifest served as static files; artwork from a Supabase-backed data layer. The "living paintings" feature is decoupled into a resumable Python pipeline, so each clip is generated once and replayed locally forever.

        

        
    
          
  • A horizontally-scrolling timeline gallery organizes the catalogue across four filename-encoded series: Old Russia, Europe, Argentina, and the United States.
    • 
          
  • Ships as an installable PWA with a web manifest, standalone display mode, and hosted app icons.
    • 
          
  • The standalone "living paintings" pipeline turns stills into looping cinemagraphs through the Kling image2video API, handling JWT auth (PyJWT), job polling (requests), and aspect ratios (Pillow).
    • 
          
  • Batch generation is resumable and skips already-rendered clips, driven by a per-painting motion-prompt manifest. A separate tooling layer (rename-map, reference-audit JSON, PowerShell rollback) handles safe bulk asset renames.
    •

          02 · Algorithmic trading / bot infrastructure
          
Freqtrade · Hyperliquid

          
A Dockerized crypto trading bot on the Hyperliquid DEX.

          
how it stays reliable hard paper/funded stack isolation — separate port + private config · 8 idempotent operator scripts behind Docker preflight

        

        

          A self-hosted deployment of the open-source freqtrade engine, configured to trade on the
          Hyperliquid DEX in paper/dry-run mode. A custom long-only strategy (EMA crossover + RSI + volume filter)
          runs on a 5-minute timeframe against a simulated wallet, with the FreqUI dashboard and REST/WS API behind Docker.
        

        

          
The problem
Hyperliquid ships no historical-OHLCV support in freqtrade, and running a bot safely demands hard separation between paper and funded execution, plus reproducible, scriptable ops.

          
The build
Two Docker Compose stacks (dry-run + a profile-gated live stack on a separate port and private config); a custom IStrategy with hand-rolled indicators; a Python importer against Hyperliquid's candleSnapshot endpoint to backfill OHLCV; and idempotent PowerShell operator scripts.

        

        
    
          
  • The custom HyperliquidDryRunStrategy combines EMA 12/26/50 trend and crossover with a hand-implemented RSI(14) entry gate and 20-period volume confirmation, long-only on spot.
    • 
          
  • Risk is codified in the strategy itself: a tiered minimal-ROI ladder, a fixed stoploss, and an offset trailing stop.
    • 
          
  • Two Docker Compose stacks isolate paper from funded execution with separate ports and a private-config split, so the two can never share state.
    • 
          
  • A custom OHLCV importer works around the exchange's missing freqtrade download-data support, pulling Hyperliquid's candleSnapshot endpoint into Feather files.
    • 
          
  • Eight idempotent PowerShell operator scripts wrap start, stop, status, logs, backtest, and pairlist checks behind Docker preflight validation.
    •

          03 · Real-time data infrastructure
          
STAKE·ODDS — Live Odds Tracker

          
A dual-feed live sports-odds tracking & movement engine.

          
how it stays reliable supervisor respawns crashed pollers · per-feed STALE freshness guard (added after a real silent-failure) · append-only snapshot + movement logs

        

        

          A Node.js pipeline that tracks live sports-betting odds across two independent books in real time. One feed
          is pulled from an authenticated GraphQL wire over a cloud browser session; the other is scraped from the live
          DOM via the Chrome DevTools Protocol. Both normalize to one model, get diffed per poll for movement,
          persisted, archived, and rendered on a dependency-free dashboard.
        

        

          
The problem
Odds move continuously and are exposed only through ephemeral, rate-limited, often-authenticated channels with no public history, and a frozen feed must never be reported as live.

          
The build
Two source-specific extractors converge on one event/outcome shape; an OddsStore diffs each poll into timestamped deltas; per-feed freshness flags a frozen book STALE independently; a supervisor respawns crashed pollers; and an archival pipeline rolls finished events into a replayable history.

        

        
    
          
  • Dual-source extraction pulls one feed from an authenticated GraphQL wire and the other from a CDP DOM evaluation, normalizing both to a single event/outcome model.
    • 
          
  • Each poll is diffed into timestamped movement deltas with direction, feeding a biggest-movers aggregation.
    • 
          
  • Per-feed freshness tracking flags a frozen book STALE on its own counter, a guard written after a real multi-hour silent-failure incident.
    • 
          
  • A token-overlap matcher pairs the same fixture across both books despite differing names (accents, club suffixes, reserve tags).
    • 
          
  • The dashboard is self-contained: inline-SVG charts, a live movements stream, and a recorded-round replay mode, all working offline.
    •

          04 · Full-stack fintech bot
          
Loansy

          
A Telegram P2P crypto-lending bot with a 60-view Flask ops console.

          
how it stays reliable conversational loan state machine · gunicorn + Docker on Railway · pytest isolated from infra via mongomock + fakeredis

        

        

          A peer-to-peer crypto-lending platform that runs as a Telegram bot backed by a large Flask admin console.
          Borrowers and lenders are matched in-chat; funds settle against on-chain USDT/USDC balances on TON and Solana;
          repayments feed a reputation/credit-score system; and operators manage the whole lifecycle from a hand-built dashboard.
        

        

          
The problem
Informal P2P lending in Telegram groups has no settlement layer, risk controls, or audit trail: lenders can't track loans, operators can't see loss exposure, and disputes are unrecoverable.

          
The build
Two coordinated services on a shared MongoDB + Redis backend: a pyTelegramBotAPI bot core driving the conversational loan state machine and on-chain wallet watchers (TON, Solana), and a Flask/Jinja admin app of 60+ templates on a single hand-built design system.

        

        
    
          
  • The Telegram bot runs the full loan lifecycle in chat: request, lender offer, accept, on-chain disbursement, repayment, and a reputation update.
    • 
          
  • On-chain custody spans two networks, with TON (pytoniq) and Solana (solders) watchers crediting deposits and reconciling repayments.
    • 
          
  • Borrowing is reputation- and tier-gated, with admin approval required on proposals.
    • 
          
  • The Flask ops console carries 60+ Jinja templates: dashboard, loans, deposits, withdrawals, disputes, risk-ops, affiliates, payouts, telemetry, and flow analytics.
    • 
          
  • It's productionized on gunicorn + Docker on Railway, with a pytest suite isolated from infra via mongomock and fakeredis.
    •

          05 · Observability / monitoring
          
ClaudeDown

          
A real-time availability monitor for AI infrastructure.

          
how it stays reliable independent edge-probe reconciled with the vendor Statuspage · tiered 5-min / 1-hour Postgres rollups · ETag conditional-GET from in-memory caches

        

        

          A status monitor that ingests Claude's public Statuspage on a fixed scan loop and independently
          edge-probes a registry of AI/dev providers, persisting availability and latency to Postgres with multi-tier
          rollups. A single-file vanilla-JS dashboard renders availability, latency, incident and component analytics,
          with hand-built inline-SVG charts and zero charting dependencies.
        

        

          
The problem
AI platforms publish status on heterogeneous pages, and there's no single view combining a provider's own status with an independent probe plus a queryable history. A page-load signal alone can't tell you whether an API is truly reachable.

          
The build
A provider registry pairs each provider with an HTTP edge probe and a Statuspage URL, reconciled into one verdict. Snapshots compact into 5-minute and 1-hour rollups so the same JSON endpoints answer 1h–30d queries; handlers serve ETag-validated JSON from in-memory caches.

        

        
    
          
  • Statuspage v2 ingestion scans 8 Claude endpoints on a 15-second loop.
    • 
          
  • A provider registry of 10 AI/dev services (OpenAI, Gemini, GitHub, Cursor, Groq, Cohere, Azure AI, Replicate, Vercel, Cloudflare) reconciles each one's status page against an independent HTTP edge probe.
    • 
          
  • The two signals collapse into a single verdict per provider: up, degraded, down, or conflict.
    • 
          
  • A tiered Postgres history compacts raw samples into 5-minute and 1-hour rollups, with configurable retention and compaction.
    • 
          
  • Conditional-GET JSON APIs with ETag/Last-Modified validators serve from in-memory caches to keep poll traffic cheap.
    • 
          
  • Every chart in the single-file dashboard is hand-rolled inline SVG: a multi-mode overlay (Swimlane, Dual-Y, Heatmap, Percentile) plus per-provider sparklines, no chart library.
    •
